# Irish Buy-to-Let Underwriting App — MVP Build Plan

## Product thesis

Build an Irish residential buy-to-let screening and underwriting app.

The app lets a user:
1. set investor assumptions once,
2. add Irish property listings manually or via URL,
3. confirm/edit extracted listing facts,
4. enter or verify rent comparables,
5. run deterministic financial modelling,
6. score risks and data quality,
7. compare properties,
8. generate a simple scorecard and full underwriting report,
9. calculate negotiation ranges and next due-diligence actions.

The MVP is Dublin-first but Ireland-capable in architecture.

Do not position this as investment advice. Use language such as screening, underwriting, scenario analysis, risk flags, and due diligence. Always include “verify with solicitor, tax advisor, broker, surveyor.”

---

# Core principles

## 1. Deterministic calculations
All financial calculations must be code-based, not LLM-generated.

The LLM may:
- extract listing facts,
- summarize listing text,
- classify rent comps,
- explain risks,
- draft narrative sections,
- draft negotiation messages.

The LLM must not be trusted to compute:
- mortgage payment,
- amortization,
- cash flow,
- tax scenarios,
- stress tests,
- ROI,
- rankings.

## 2. User confirmation
Every auto-extracted property fact must pass through a user confirmation/edit screen before analysis.

## 3. Data quality is first-class
Every property must show:
- confirmed fields,
- estimated fields,
- unknown fields,
- source/evidence,
- data quality score,
- ranking confidence.

## 4. Dublin-first, Ireland-capable
Initial calibration is Dublin-focused. Non-Dublin properties can be entered, but outputs must show lower confidence until market-specific calibration is added.

## 5. No monitoring in MVP v1
Do not build scheduled monitoring, Gmail ingestion, or email alerts in v1. Add later after core underwriting workflow is stable.

2. Safety / tenant perception

7. Building quality

12. Layout/size quality
13. Parking value
14. Resale liquidity
15. Operational hassle

## Data Quality Score
Start at 10.
Deduct for:
- size unknown,
- service charge estimated/unknown,
- BER unknown,
- tenancy unknown,
- micro-location uncertain,
- previous rent unknown where relevant,
- only weak rent comps.

## Acceptance criteria
- Scores render with explanation.

- BER D/E produces capex/risk flags.
- Unknown tenancy reduces legal rent confidence.
- Estimated service charge triggers service-charge sensitivity.
Phase 4 — Rent comp module
# Phase 4 — Rent Comparables

## Goal
Allow users to add rent comps manually and classify relevance.

## UI
For each rent comp:
- URL/source,
- address/development,
- rent,
- beds,
- baths,
- size,
- BER,
- parking,
- same building? yes/no,
- comp tier A/B/C/D,
- adjustment notes.

## Logic
Comp tiers:
- Tier A: same building/block
- Tier B: same development/adjacent block
- Tier C: same micro-area
- Tier D: broader area

Rules:
- High rent confidence requires Tier A or B.
- Medium requires multiple strong Tier C comps.
- Tier D only means Low confidence.
- If no comps are entered, rent confidence is Low.

## Outputs
- conservative rent,
- base rent,
- strong rent,
- underwriting rent,
- rent confidence,
- market rent vs legal rent distinction.

## Acceptance criteria
- User can add comps.
- App shows rent confidence.
- Weak comp evidence reduces ranking confidence.
- Rent sensitivity table appears for top properties.
Phase 5 — Negotiation engine
# Phase 5 — Negotiation Engine

## Goal
Implement price sensitivity and offer strategy.

## Package
`packages/negotiation-engine`

## Inputs
- asking price,
- financial model,
- stress results,
- risks,
- listing age,
- data quality,
- OMC/RTB/BER flags,
- user negotiation aggressiveness.

## Outputs
- price sensitivity table,
- opening offer,
- target price,
- stretch price,
- walk-away,
- acceptance probability,
- DD findings that increase/decrease offer,
- agent message.

## Rules
- Use 30% tax central case.
- Offer amounts rounded to nearest €500/€1,000.
- No false precision.
- Aggressive offers require evidence: stale listing, BER issue, OMC risk, weak stress, rent uncertainty.
- New clean listings should not assume large discount.
- OMC unresolved and RTB unresolved should lower offer or trigger conditional/no-offer.

## Acceptance criteria
- Price sensitivity generated at asking, -2.5%, -5%, -7.5%, -10%.
- Offer amounts are rounded.
- Walk-away price is logical and consistent across UI.
- Agent script is generated from real risks only.

Definition of MVP success

The MVP is successful when:

A user can create an investor profile.
A user can enter or import 5 Dublin properties.
The app calculates mortgage, cash flow, ROI, tax sensitivity, and stress tests correctly.
The app distinguishes confirmed, estimated, and unknown data.
The app produces data quality scores.
The app ranks properties by yield, resilience, and overall risk-adjusted attractiveness.
The app generates realistic negotiation ranges.
The app exports a shareable report.
The app avoids the known failure modes:
wrong location label,
missing size,
weak rent comps treated as strong,
BER treated as legal rent cap,
20% tax treated as central non-resident case,
OMC called clean without documents,
unconfirmed properties ranked with confirmed properties,
overly precise negotiation numbers,
high yield over-ranked despite weak evidence.

If all nine are true, v1 is ready for private user testing.

## Final improved framing

The product you are building is not primarily a report generator. It is a **decision-control system** for property investors.

The core promise should be:

> “Stop relying on gut feel. Underwrite every Irish property the same way: rent evidence, mortgage burden, tax sensitivity, stress resilience, legal/OMC/BER risks, and negotiation price.”

For MVP, the killer use case is:

> “Paste or enter five Dublin property listings and get a ranked, auditable buy-to-let shortlist showing what is worth viewing, what is risky, what price makes each deal work, and what evidence must be verified before offering.”

That is specific enough to build, test, and improve sequentially.
